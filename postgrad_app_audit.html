二、后端逻辑与接口问题

级别 问题 表现 / 影响 建议
高推荐结果快照字段与前端展示不完全一致历史详情页与结果页依赖不同的数据结构，部分字段名使用驼峰/下划线混杂，容易出现页面某些列空白、标签丢失或回显不一致的问题。统一推荐结果 DTO/JSON schema，前后端只使用一套标准字段命名。
高收藏、历史、详情接口返回结构依赖隐式字段例如历史页直接展示 rule_version、created_at，若后端返回为驼峰命名就会空白；同类问题在收藏页也可能出现。后端统一响应序列化策略，前端增加兜底格式化。
高结果分页/分组逻辑可能造成用户误解推荐结果按组切分后只展示前若干条，用户以为“没有更多”，但实际上只是分页截断；同时结果页存在“展开全部”按钮但未实现真正展开。明确标识“当前仅展示前 N 条”，或补充分页/加载更多接口。
高后端允许依赖客户端传入的评分与筛选条件预计分数、专业方向、地区等均从请求体传入，若缺少校验或风控策略，用户可构造异常请求，得到不符合常识的推荐结果。对关键参数做范围校验、类型校验和业务白名单限制。
中推荐结果缓存依赖 sessionStorage页面切换后可能读取到旧数据；当用户重新生成推荐但未刷新时，历史和当前结果容易串线。缓存应携带版本号或 recommendationId，并在进入结果页时优先以服务端为准。
中API 错误处理偏弱多个页面只做了 then/finally，缺少统一的 catch 提示与兜底状态，接口失败时页面可能仅停留在 loading 或空白区域。在页面层增加统一错误提示、空态和重试按钮。
中历史详情展示和结果详情存在重复逻辑历史页与结果页分别解析推荐 payload，若后端结构变化会维护两处，容易不一致。抽出统一的数据适配层或公共展示组件。
中旧“学习方式”字段仍可能残留当前已决定隐藏该维度，但若后端、缓存或历史数据里仍保留字段，容易让数据含义混乱。彻底删除后端接口、SQL、前端展示、缓存中的相关字段。
低部分接口未显式标注参数默认值例如历史/详情接口在前端看不出必填与选填边界，后续维护容易误传参数。补充接口文档与参数说明。
低推荐等级/分组命名不统一页面中有“冲刺/稳中偏冲/稳妥候选/保底候选”，历史页又是“稳妥/重点关注/可冲/数据不足”，语义映射不一致。统一分组命名和映射规则。

三、安全问题

级别 问题 风险说明 建议
高Token 可能存放于浏览器可读存储当前前端通过本地存储读取 token，如果页面存在 XSS 漏洞，攻击脚本可直接读取并冒用用户身份。改用 HttpOnly Cookie 或增加更严格的 CSP、防 XSS 处理。
高推荐结果包含敏感画像信息画像、目标地区、分数、风险偏好等会被保存为历史记录；若权限校验不严，可能被越权查看。历史记录接口必须做用户归属校验与最小字段返回。
高前端内容未体现输出编码策略多处将后端返回的字符串直接渲染到页面，如果后端数据源污染，可能放大 XSS 风险。后端输出净化，前端避免使用不安全的 HTML 渲染方式。
中仅依赖前端登录态控制收藏操作未登录时前端允许“假收藏”，登录后再同步，容易让用户误以为已成功持久化。未登录时应明确提示并阻止持久化动作，或提供清晰的临时态说明。
中接口缺少明显的限流与审计痕迹生成推荐、查看详情、列表接口可能被批量刷取，带来数据爬取与资源消耗风险。对核心接口加限流、监控和审计。
低外链跳转未进行站点白名单管理部分页面直接打开来源链接，虽然当前是 N诺来源，但长期看可能引入不安全外链。统一外链白名单或提示二次确认。

四、前端 UI 与交互问题

级别 问题 表现 建议
高结果页存在静态“占位式”按钮“导出结果”“调整列”“一键对比”等按钮看起来像真实功能，但若未接通实际逻辑，会给用户造成误导。未实现时请禁用或隐藏，并标注“开发中”。
高部分关键空态处理不足接口失败、无数据、结果过少等状态没有统一的视觉反馈，用户容易认为页面卡死。增加空状态、错误状态与重试入口。
中结果页“推荐结果/对比与备选”切换逻辑略混乱页面通过 query 和内部状态共同控制，若刷新、回退或直接分享链接，tab 状态容易不一致。将 tab 状态完全由路由 query 驱动，减少本地状态分叉。
中筛选器布局在中小屏较拥挤左侧筛选栏在宽屏下较好，但移动端/平板下会变成纵向堆叠，信息密度高且操作疲劳。把筛选面板改为折叠抽屉或顶部弹层。
中历史页、收藏页表格字段过宽专业名称、学院名称、代码、学位等列很多，窄屏时横向滚动体验差。支持列配置、主次信息分层展示。
低说明文案重复且偏长风险提示、N诺说明在多个页面反复出现，用户会产生“信息噪音”。合并为统一的提示组件或侧栏说明卡片。
低视觉风格局部不统一一些页面偏“营销风”，另一些页面偏“管理系统风”，字体大小、按钮风格与阴影层次有差异。统一设计语言与组件规范。
低图标与状态提示不够明确收藏、加载、结果说明、警告图标较多，但缺少统一状态语义，易混淆。制定图标语义表与颜色规范。

五、可维护性与代码质量问题

  • 数据适配逻辑分散。 推荐页、结果页、历史页、收藏页都在各自处理字段映射，建议抽出统一的适配器。
  • 重复提示文案过多。 例如“复试线不是最低录取分”“N诺数据仅供参考”等多处重复，后期改文案成本高。
  • 部分页面缺少统一错误边界。 异步接口失败后没有通用降级策略。
  • 旧功能残留较多。 例如被移除的学习方式相关字段，前后端、缓存、历史数据可能还残留，需要一次性清理。
  • 接口与 UI 耦合较重。 页面直接依赖后端字段名和枚举值，导致后端小改动即可引发前端展示问题。

六、建议整改顺序

  1. 先修数据链路：统一推荐结果、历史记录、收藏详情的字段命名与结构，消除前后端不一致。
  2. 再修安全：收紧 token 存储方式、越权校验、接口限流与审计。
  3. 然后修交互：补齐空态、错误态、按钮真实能力与加载反馈。
  4. 最后做体验优化：统一视觉规范、合并说明文案、优化中小屏布局。
说明：本报告基于当前仓库可见代码静态审查生成，未运行完整端到端测试；部分问题属于“高概率风险”或“设计缺陷”，需要结合实际运行进一步确认。
