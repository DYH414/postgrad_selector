# 需求文档 — AI推荐报告重构

## 1. 业务流程分析

### 1.1 背景和目标

这是个人项目中的 AI 推荐报告功能重构，不是大型平台级重建。本次目标是把现有 AI 推荐报告流程梳理清楚，废弃旧流程中含糊的“AI 推荐、加入报告、自动补充、最终报告生成”机制，重新建立一个简单、可控、可解释的推荐链路。

本次只重构 AI 推荐报告功能，不重构规则筛选功能。规则筛选仍按原有方式工作。

核心目标：

1. 用户画像已有时，用户点击一次即可生成 AI 推荐草稿。
2. 推荐结果默认按冲刺、稳妥、保底展示，目标数量为 3/4/3，但不强制补满。
3. AI 只能在后端给定的候选池和事实卡范围内选择学校，不编造事实。
4. 右侧报告草稿就是最终报告的来源，最终报告不再重新偷偷选择学校。
5. 用户可以移除、加回或替换候选。
6. 候选不足时如实展示，不为了凑满 10 所学校而降低可信度。
7. 旧 AI 推荐报告流程准备废弃，新流程不需要兼容旧流程语义。

### 1.2 关键输入输出

| 阶段 | 输入 | 输出 | 说明 |
|------|------|------|------|
| 生成草稿 | 用户画像、本地候选数据、生成指令 | 推荐草稿 | 用户点击后生成冲刺/稳妥/保底候选 |
| AI 取舍 | 候选事实卡、分档结果 | AI 选择的候选和理由 | AI 只做选择和解释，不生成事实 |
| 用户调整 | 当前草稿、用户点击或聊天指令 | 更新后的草稿 | 支持移除、加回、替换、补充 |
| 生成报告 | 当前右侧草稿、画像、候选事实、AI 理由 | 最终报告 | 最终报告必须和草稿一致 |
| 查看报告 | 历史报告 | 报告详情 | 展示生成时的结果，不随当前数据自动变化 |

### 场景 1：生成初始推荐草稿

触发条件：用户进入 AI 推荐页，点击“生成 AI 推荐草稿”。

前置条件：

- 用户已登录。
- 用户画像至少有预计分数。
- 用户画像已有专业方向或默认使用 408 计算机相关范围。

输入：

| 输入项 | 必填 | 说明 |
|--------|------|------|
| 用户画像 | 是 | 预计分数、地区偏好、风险偏好、学校层次偏好等 |
| 本地候选数据 | 是 | 学校、专业方向、近三年分数、招生人数、录取情况等 |
| 生成指令 | 是 | 用户主动点击生成 |

主流程：

1. 用户点击“生成 AI 推荐草稿”。
2. 系统根据用户画像筛出候选池。
3. 系统为候选整理事实卡。
4. 系统将候选划分为冲刺、稳妥、保底或待核验。
5. AI 在给定候选事实卡内选择推荐候选，并给出理由和风险。
6. 系统校验 AI 结果，去掉候选池外、重复、数据不足的候选。
7. 系统生成右侧报告草稿。

输出：

| 输出项 | 说明 |
|--------|------|
| 右侧报告草稿 | 按冲刺、稳妥、保底展示候选 |
| 推荐理由 | AI 对每个候选的选择理由 |
| 风险提示 | 每个候选的主要风险 |
| 候选不足说明 | 不满 3/4/3 时说明原因 |

异常流程：

- 用户画像缺少预计分数：提示先补充预计分数。
- 候选池为空：提示当前条件下没有可信候选。
- AI 返回候选池外内容：系统丢弃，不进入草稿。
- 某档候选不足：如实展示，不强行补满。

### 场景 2：调整报告草稿

触发条件：用户在右侧草稿中操作，或在聊天中明确表达调整意图。

输入：

| 输入项 | 必填 | 说明 |
|--------|------|------|
| 当前报告草稿 | 是 | 当前已选候选 |
| 用户调整动作 | 是 | 移除、加回、替换、补充 |
| 用户聊天文本 | 否 | 如“不要这个学校”“换一个更稳的” |

主流程：

1. 用户点击“移出报告”，或在聊天中明确表达调整意图。
2. 系统判断该动作是否会改变草稿。
3. 如果只是询问或分析，不改变草稿。
4. 如果是移除、加回、替换或补充，则更新右侧草稿。
5. AI 回复本次调整结果和原因。

输出：

| 输出项 | 说明 |
|--------|------|
| 更新后的草稿 | 右侧候选同步变化 |
| 调整说明 | 告诉用户移除了谁、加回了谁、替换成谁 |
| 无法调整原因 | 如指代不清、没有更合适候选、数据不足 |

异常流程：

- 聊天指代不清：不改草稿，要求用户明确学校或档位。
- 用户移除后某档不足：不自动补齐。
- 用户要求加回的数据不足候选：提示原因，不直接进入主报告。

### 场景 3：生成最终报告

触发条件：用户点击“生成最终报告”。

前置条件：

- 右侧报告草稿中至少有 1 个可信候选。

输入：

| 输入项 | 必填 | 说明 |
|--------|------|------|
| 当前右侧草稿 | 是 | 最终报告唯一来源 |
| 用户画像 | 是 | 用于说明推荐依据 |
| 候选事实卡 | 是 | 用于展示事实依据 |
| AI 推荐理由 | 是 | 用于展示解释和风险 |

主流程：

1. 用户点击“生成最终报告”。
2. 系统读取当前右侧报告草稿。
3. 系统按草稿内容生成最终报告。
4. 报告按冲刺、稳妥、保底展示。
5. 如果不满 3/4/3，报告中说明原因。

输出：

| 输出项 | 说明 |
|--------|------|
| 最终报告 | 与右侧草稿候选一致 |
| 推荐摘要 | 说明本次策略和候选数量 |
| 分档结果 | 冲刺、稳妥、保底候选 |
| 不足说明 | 候选不足或数据不足时展示 |

异常流程：

- 草稿为空：提示先生成草稿。
- 某档为空：允许生成报告，并显示该档暂无可信候选。

### 场景 4：查看历史报告

触发条件：用户打开已生成的 AI 推荐报告。

输入：

| 输入项 | 必填 | 说明 |
|--------|------|------|
| 用户身份 | 是 | 只查看自己的报告 |
| 历史报告 | 是 | 已生成的报告 |

主流程：

1. 用户打开历史报告。
2. 系统展示生成时的推荐结果。
3. 报告内容保持生成时状态，不因当前数据变化自动改写。

输出：

| 输出项 | 说明 |
|--------|------|
| 历史报告正文 | 展示当时生成的推荐结果 |
| 画像依据 | 展示当时用于推荐的用户画像摘要 |
| 候选事实 | 展示当时使用的数据依据 |

## 3. 用户故事 + 验收标准

### US-001：生成推荐草稿

作为考研用户，我想一键生成 AI 推荐草稿，以便快速看到适合我的择校组合。

验收标准：

- AC1：给定用户画像包含预计分数，当用户点击生成草稿，则系统生成右侧报告草稿。
- AC2：给定某档候选不足，当草稿生成完成，则该档可以少于目标数量。
- AC3：给定 AI 返回候选池外候选，则该候选不会出现在草稿中。
- AC4：给定候选数据不足，则该候选不得进入冲刺、稳妥、保底主档。

### US-002：理解推荐结果

作为考研用户，我想看到每个候选为什么被推荐，以便判断是否接受。

验收标准：

- AC1：给定草稿已生成，用户能看到候选所属档位。
- AC2：用户能看到候选的事实摘要、推荐理由和风险提示。
- AC3：用户能区分事实依据和 AI 观点。

### US-003：调整报告草稿

作为考研用户，我想移除、加回或替换候选，以便报告只保留我愿意考虑的学校。

验收标准：

- AC1：用户点击“移出报告”后，该候选从草稿中消失。
- AC2：用户只是询问候选情况时，草稿不发生变化。
- AC3：用户明确要求替换时，系统给出替代候选或说明没有合适候选。
- AC4：用户移除后某档不足，系统不自动补齐。

### US-004：生成最终报告

作为考研用户，我想把当前草稿生成最终报告，以便保存完整择校方案。

验收标准：

- AC1：最终报告候选必须和生成前右侧草稿一致。
- AC2：草稿不满 3/4/3 时仍可生成报告。
- AC3：某档为空时，报告显示暂无可信候选，不编造学校。
- AC4：报告中展示本次推荐依据和不足原因。

### US-005：查看历史报告

作为考研用户，我想查看历史 AI 推荐报告，以便回顾当时的择校判断。

验收标准：

- AC1：用户能打开自己已生成的历史报告。
- AC2：历史报告展示生成时的推荐结果。
- AC3：当前数据变化时，历史报告正文不被自动改写。

### US-006：废弃旧 AI 推荐报告流程

作为产品负责人，我想废弃旧 AI 推荐报告流程，以便新功能按清晰的新草稿机制运行。

验收标准：

- AC1：用户生成新 AI 推荐报告时，进入新草稿到最终报告流程。
- AC2：最终报告以右侧草稿为准，不沿用旧流程中的隐式加入或自动补充语义。
- AC3：旧历史报告如需保留，仅用于查看，不再用于生成新报告。

## 4. 非功能需求（NFR）

| 编号 | 类别 | 要求 | 量化标准 |
|------|------|------|----------|
| NFR001 | 真实性 | AI 不得生成事实字段 | 学校、专业、分数、招生人数、年份等事实均来自系统数据 |
| NFR002 | 可解释性 | 推荐必须有理由 | 每个主候选至少展示事实摘要、推荐理由、风险提示 |
| NFR003 | 一致性 | 草稿和最终报告一致 | 最终报告候选必须来自生成时右侧草稿 |
| NFR004 | 容错 | 不强行凑满 3/4/3 | 候选不足时允许生成报告并说明原因 |
| NFR005 | 可用性 | 操作简单 | 用户从进入页面到看到草稿不超过 2 个主要操作 |
| NFR006 | 等待反馈 | 生成过程有状态提示 | 生成草稿时用户能看到明确的等待或处理中状态 |
| NFR007 | 废弃边界 | 旧流程不作为兼容目标 | 新报告生成入口和报告语义以本需求为准 |

## 6. 待确认问题

Q1：专业方向缺失时是否允许生成草稿？

- 选项 A：允许，默认使用 408 计算机相关范围。
- 选项 B：不允许，要求用户先补充专业方向。
- 建议：选 A，个人项目先减少阻断。

Q2：同一学校多个专业方向是否允许同时进入报告？

- 选项 A：允许，但尽量避免同一学校占太多位置。
- 选项 B：不允许，同一学校只保留一个方向。
- 建议：选 A，因为不同学院和专业方向难度可能不同。

Q3：候选池只有 5 所学校时，是否强行补到 10 所？

- 选项 A：不补满，只展示 5 所可信候选。
- 选项 B：放宽条件补满。
- 建议：选 A，符合推荐真实性。

Q4：旧历史报告是否继续可查看？

- 选项 A：可以查看，但不再用于新报告生成。
- 选项 B：不再展示旧报告。
- 建议：选 A，避免用户历史资产突然消失。
