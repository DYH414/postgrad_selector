# 平均分下探筛选规则设计

## 背景

用户端快速推荐页当前有一个分数范围控件。此前它容易被理解成围绕用户预计分上下浮动，或围绕最低录取分/最高录取分扩展。新的产品口径明确为：用拟录取平均分作为主判断依据，让用户按自己的预计初试总分寻找“刚好够线、可以冲一点”的院校专业。

## 目标

当用户选择 `scoreRange=N` 时，系统返回预计分达到“拟录取平均分下探 N 分”的院校专业，并按风险从高到低排序。这里的风险从高到低指更接近下探边界的项目更靠前，方便用户先看到更有冲刺价值的候选。

## 非目标

本次不重新设计完整冲稳保算法，不引入新的外部数据源，不改变收藏、历史、登录等用户链路。最低录取分和最高录取分继续用于结果解释，不作为本筛选模式的主进入条件。

## 规则

输入字段保持为 `scoreRange`，但语义调整为“平均分下探”：

```text
预计初试总分 >= 拟录取平均分 - scoreRange
```

例子：

```text
预计分 = 300
scoreRange = 15
可匹配：拟录取平均分 <= 315
不可匹配：拟录取平均分 > 315
```

当 `scoreRange` 为 `null` 时，系统不启用平均分下探筛选，保持常规推荐分组逻辑。

## 数据质量

主结果必须有拟录取平均分。没有 `avg_admitted_score` 的院校专业不进入平均分下探主结果。这样可以避免用最低分、复试线或缺失数据猜测录取难度。

## 排序

平均分下探模式下按“风险从高到低”排序：

```text
avgScoreGap = 预计分 - 拟录取平均分
```

在已满足 `avgScoreGap >= -scoreRange` 的候选里，越接近 `-scoreRange` 越靠前。

示例，`scoreRange = 15`：

```text
avgScoreGap = -14  优先展示，偏冲
avgScoreGap = -3   次之
avgScoreGap = +20  靠后，更稳
```

## 后端设计

`RecommendationMapper.selectCandidates` 在 `scoreRange != null` 时增加平均分条件：

```sql
ar.avg_admitted_score is not null
and #{estimatedScore} >= cast(ar.avg_admitted_score as signed) - #{scoreRange}
```

排序优先按下探边界接近程度：

```sql
abs((cast(ar.avg_admitted_score as signed) - #{scoreRange}) - #{estimatedScore}) asc
```

服务层继续把 `scoreRange` 写入 request snapshot。返回结构仍使用单一 `matches` 分组，保持前端已有平铺模式。

## 前端设计

推荐页控件文案调整：

- 控件名：`平均分下探`
- 按钮：`均分-5`、`均分-10`、`均分-15`、`均分-20`、`不限`
- 帮助说明：解释这是用用户预计分和拟录取平均分比较，不是复试线，也不是最低录取分。

结果页展示调整：

- 顶部 chip：`平均分下探 15分`
- 平铺结果说明：`估分达到拟录取均分下探范围`
- 卡片优先展示 `拟录取均分` 和 `均分差距`
- 保留 `最低录取分`、`拟录取区间` 作为风险辅助信息

## 错误和空状态

如果平均分下探模式没有匹配结果，显示空状态：

```text
没有符合平均分下探条件的院校，请放宽下探分数或调整筛选条件。
```

如果候选缺平均分，不进入主结果，不在前端伪造平均分或用最低录取分替代。

## 测试

后端单测覆盖：

- 有平均分且满足 `estimatedScore >= avgScore - scoreRange` 时进入 `matches`
- 有平均分但不满足时不进入
- 缺平均分时不进入平均分下探主结果
- 排序时更接近下探边界的候选排在前面
- `scoreRange = null` 时保持常规冲稳保分组

前端验证覆盖：

- 推荐页显示 `平均分下探` 和 `均分-N`
- 帮助弹窗解释平均分下探规则
- 结果页 chip 和空状态文案正确
- 构建通过
