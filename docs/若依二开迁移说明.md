## 4. 用户体系规划

后续开发建议保留两套用户体系：

- `sys_user`：若依自带后台管理员用户，用于后台登录、菜单、权限和代码生成器。
- `app_user`：考研择校普通用户，用于同学朋友注册登录、画像、收藏和推荐历史。

不要把普通用户直接混进若依后台用户表，否则权限边界会变复杂。

## 5. 二开开发顺序

建议后续按以下顺序推进：

1. 先导入若依系统 SQL，并启动后端和前端。
2. 导入考研择校业务 SQL。
3. 使用若依代码生成器，为基础业务表生成后台 CRUD。
4. 优先生成并调整以下后台模块：
   - 学校管理：`school`
   - 学院管理：`college`
   - 专业方向管理：`program`
   - 科目管理：`subject`
   - 专业科目管理：`program_subject`
   - 复试线管理：`admission_score`
   - 招生计划管理：`admission_plan`
   - 拟录取统计管理：`admission_result`
   - 数据来源管理：`data_source`
   - 数据完整度管理：`program_year_data_quality`
5. 再实现普通用户端接口：
   - 注册登录
   - 用户画像
   - 推荐结果
   - 收藏专业方向
   - 历史推荐记录
6. 最后接入 Agent 辅助能力：
   - 数据采集任务
   - staging 暂存审核
   - 推荐解释
   - 报告生成

## 7. 注意事项

- 若依后台启动依赖 MySQL 和 Redis。
- 若依默认后台账号通常来自 `ry_20260417.sql` 初始化数据。
- 业务表导入后，暂时不会自动出现在菜单中，需要通过代码生成器生成代码并配置菜单。
- 业务普通用户表 `app_user` 后续需要单独实现注册登录接口，不建议直接复用 `sys_user`。
- Agent 后续产出的数据必须先进入 `staging` 或待审核流程，不能直接写入主表。
