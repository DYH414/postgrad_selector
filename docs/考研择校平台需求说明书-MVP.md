# 考研择校平台需求说明书（MVP版）

版本：1.0  
日期：2026-05-18  
适用阶段：单人开发 MVP  
主要开发者：正在准备考研的大学生开发者  
主要技术栈：Spring Boot、MySQL、MyBatis-Plus、Vue

## 1. 文档目的

本文档用于指导考研择校平台 MVP 阶段的功能开发、数据库使用、页面设计和验收测试。

本项目第一阶段不是一次性建设完整商业平台，而是先做出一个可给同学朋友试用的择校推荐系统。用户可以注册登录，填写自己的考研画像，系统根据已维护的学校、专业、历年复试线、招生计划、拟录取数据和数据来源，生成推荐学校专业方向列表，并展示推荐档位、风险提示和数据可信度。

本需求说明书面向开发使用，重点说明第一版必须实现什么、暂时不做什么、每个模块依赖哪些数据表，以及功能完成后如何验收。

## 2. 项目背景

考研择校信息分散在研招网、学校研究生招生网、学院官网、招生目录、复试细则、拟录取名单、第三方资料站和经验贴中。普通考生在择校时经常遇到以下问题：

- 容易漏掉适合自己的学校、学院或专业方向。
- 只看复试线，忽略拟录取最低分、招生人数和复录比。
- 很多推荐只给结论，不给数据来源。
- 不同分数段、地区偏好和风险偏好的用户，对同一学校的适配程度不同。
- 手工整理学校信息成本高，且数据更新后不容易维护。

本项目希望通过结构化数据、规则推荐和来源证据链，帮助考生更高效地完成择校初筛。

## 3. 产品定位

MVP 阶段产品定位为：

```text
面向考研学生的轻量级择校推荐工具。
```

第一版优先服务同学朋友试用场景。系统需要具备基础账号能力、用户画像填写、规则推荐、学校专业详情、收藏、历史推荐记录和管理后台数据维护能力。

系统长期可以扩展为全专业、全国性考研择校平台，并接入 Agent 做推荐解释、数据缺口发现、资料采集辅助和报告生成。但 MVP 阶段推荐结果应先由明确、可控、可调试的规则产生，不依赖 Agent 直接做最终决策。

- 用户查看学校专业详情。

- 初试科目管理。
- 专业考试科目维护。
- 复试线管理。
- 招生计划管理。
- 拟录取统计管理。
- 数据来源管理。

### 4.2 MVP 暂不实现

- 在线支付。
- 会员订阅。
- 完整 PDF/Excel 报告导出。
- 自动爬虫全量采集。
- Agent 自动入库。
- 复杂机器学习推荐模型。
- 向量数据库或 RAG 问答。
- 多角色复杂权限系统。
- 短信验证码。
- 微信登录。
- 高并发部署。
- 机构端管理。

这些能力可作为二期或三期扩展。

## 5. 用户角色

### 5.1 普通用户

普通用户是考研学生或试用同学。普通用户可以注册账号、填写画像、生成推荐、查看学校专业详情、收藏专业方向和查看历史推荐记录。

普通用户不能修改学校、专业、分数线、招生计划、拟录取数据和推荐规则。

### 5.2 管理员

管理员是项目开发者本人。管理员负责维护基础数据、招生数据、数据来源、数据完整度和推荐规则。

管理员可以通过后台手动录入或修改数据。后续接入 Agent 后，Agent 抽取结果也应先进入待审核状态，由管理员确认后再写入主数据表。

## 6. 总体业务流程

### 6.1 普通用户推荐流程

```text
用户注册/登录
-> 填写考研画像
-> 选择目标专业和目标地区
-> 输入预估分数和风险偏好
-> 点击生成推荐
-> 系统召回候选专业方向
-> 系统计算推荐档位和风险提示
-> 保存推荐结果快照
-> 用户查看推荐列表
-> 用户查看详情或收藏专业方向
```

### 6.2 管理员数据维护流程

```text
管理员登录后台
-> 维护学校、学院、专业、科目基础数据
-> 维护复试线、招生计划、拟录取统计
-> 绑定数据来源
-> 标记数据可信度和完整度
-> 需要补充的数据生成采集任务
-> 推荐系统使用已审核数据
```

### 6.3 后续 Agent 辅助流程

```text
管理员创建数据采集任务
-> Agent 根据任务寻找或解析公开资料
-> Agent 输出结构化候选结果
-> 候选结果进入 staging
-> 管理员审核
-> 审核通过后写入主库
-> 推荐结果重新计算
```

MVP 阶段可先保留数据结构和页面入口，Agent 具体自动化能力不作为第一版验收条件。

## 7. 功能需求

## 7.1 账号与登录模块

### 7.1.1 用户注册

功能说明：

普通用户可以通过邮箱或手机号注册账号。MVP 阶段可以优先支持邮箱注册，手机号字段可作为后续扩展。

输入：

- 邮箱或手机号。
- 密码。
- 确认密码。
- 是否同意隐私政策。

输出：

- 注册成功后创建用户账号。
- 注册失败时返回明确错误信息。

业务规则：

- 密码必须加密存储，不能保存明文。
- 同一个邮箱或手机号不能重复注册。
- 用户必须同意隐私政策后才能注册。
- MVP 阶段可以不接入邮箱验证码或短信验证码。

数据表：

- `app_user`

验收标准：

- 可以成功注册普通用户。
- 重复邮箱或手机号不能注册。
- 未同意隐私政策不能注册。
- 数据库中只保存密码哈希。

### 7.1.2 用户登录

功能说明：

用户输入账号和密码后登录系统。

输入：

- 邮箱或手机号。
- 密码。

输出：

- 登录成功后返回登录凭证。
- 登录失败时提示账号或密码错误。

业务规则：

- 普通用户只能访问用户端功能。
- 管理员可以访问管理后台。
- 登录状态过期后需要重新登录。

数据表：

- `app_user`

验收标准：

- 普通用户登录后可以进入用户端。
- 管理员登录后可以进入后台。
- 错误密码不能登录。

### 7.1.3 退出登录

功能说明：

用户可以主动退出当前登录状态。

验收标准：

- 退出后不能继续访问需要登录的接口。

## 7.2 用户画像模块

### 7.2.1 填写用户画像

功能说明：

用户填写自己的考研基础信息，系统后续基于这些信息生成推荐结果。

必填字段：

- 初试预估总分。
- 目标地区。
- 风险偏好。
- 是否接受非全日制。
- 是否接受调剂。
- 是否接受学硕。
- 是否接受联培或中外合作。
- 目标专业代码。

选填字段：

- 本科院校层次。
- 本科专业。
- 是否跨考。
- 数学基础。
- 英语基础。
- 专业课基础。
- 每日学习时间。
- 当前复习进度。

输出：

- 保存用户画像。
- 保存用户目标专业代码。

业务规则：

- 每个用户默认维护一份当前画像。
- 用户可以修改画像。
- 目标地区可以选择多个省份或城市。
- 目标专业代码可以选择多个。
- 预估分数必须大于 0，且不能超过对应考试满分。

数据表：

- `user_profile`
- `user_target_program`

验收标准：

- 用户可以新增和修改自己的画像。
- 修改画像后再次推荐应使用最新画像。
- 非本人不能查看或修改该画像。

## 7.4 推荐模块

### 7.4.1 候选专业召回

功能说明：

系统根据用户画像召回可能适合的专业方向。

召回条件：

- 目标地区匹配。
- 目标专业代码匹配。
- 学习方式匹配。
- 学位类型匹配。
- 是否接受联培或中外合作。
- 专业状态为可用。
- 考试类型和分数满分匹配。

业务规则：

- 用户不接受非全日制时，过滤 `part_time`。
- 用户不接受学硕时，过滤 `academic`。
- 用户不接受联培或中外合作时，过滤相关方向。
- 用户设置目标地区时，只召回目标地区内学校。
- 第一阶段可以重点支持 408 计算机相关方向。

数据表：

- `user_profile`
- `user_target_program`
- `school`
- `college`
- `program`
- `program_subject`
- `subject`

验收标准：

- 输入相同画像时，候选召回结果稳定。
- 修改地区或专业代码后，召回结果随之变化。

### 7.4.2 有效难度分计算

功能说明：

系统不能只用复试线判断难度，需要结合拟录取最低分计算有效难度分。

MVP 初始规则：

```text
effective_score = max(近三年复试线中位数, 近三年拟录取最低分中位数)
```

降级规则：

```text
如果拟录取最低分缺失，则使用近三年复试线中位数。
如果复试线和拟录取数据都严重缺失，则标记为数据不足。
```

数据表：

- `admission_score`
- `admission_result`
- `program_year_data_quality`

验收标准：

- 有完整数据时，可以计算有效难度分。
- 拟录取最低分缺失时，可以降级使用复试线。
- 关键数据不足时，不强行给出确定性推荐档位。

### 7.4.3 推荐档位判断

功能说明：

系统根据用户预估分和有效难度分生成推荐档位。

初始规则：

```text
score_gap = 用户预估分 - effective_score

score_gap >= 20：重点稳妥
score_gap >= 5 且 < 20：重点关注
score_gap >= -10 且 < 5：略冲
score_gap < -10：不建议
关键数据缺失：数据不足
```

数据表：

- `recommendation_rule`
- `recommendation_log`

验收标准：

- 推荐档位与规则配置一致。
- 推荐结果中应展示档位依据。
- 数据不足时应展示数据不足原因。

### 7.4.4 风险提示

功能说明：

系统对推荐结果进行风险修正和提示。

风险类型：

- 招生人数过少。
- 拟录取最低分长期高于复试线。
- 复试线波动较大。
- 复录比偏高。
- 不保护一志愿。
- 数据完整度低。
- 关键数据来自第三方或待核验。

业务规则：

- 招生人数少于规则阈值时，提示招生规模小。
- 拟录取最低分高于复试线较多时，提示复试线参考价值有限。
- 数据完整度为 D 或 E 时，推荐结果应标记为数据不足或谨慎参考。

数据表：

- `admission_plan`
- `admission_result`
- `admission_score`
- `program_year_data_quality`
- `recommendation_rule`

验收标准：

- 推荐结果中能看到风险提示。
- 风险提示不能空泛，必须对应具体数据原因。

### 7.4.5 保存历史推荐记录

功能说明：

用户每次生成推荐结果后，系统保存本次画像快照和推荐结果快照。

保存内容：

- 用户 ID。
- 画像快照。
- 推荐结果 JSON。
- 规则版本。
- 数据版本。
- 是否付费报告。
- 生成时间。

数据表：

- `recommendation_log`

验收标准：

- 用户可以查看自己的历史推荐记录。
- 历史记录不受用户后续修改画像影响。
- 用户不能查看其他人的历史推荐记录。

## 7.5 收藏模块

### 7.5.1 收藏专业方向

功能说明：

用户可以收藏自己关注的专业方向，便于后续对比和查看。

收藏对象：

- 具体 `program`，即某学校某学院下的某个专业方向。

输入：

- 专业方向 ID。
- 用户备注，可选。

输出：

- 收藏成功。

业务规则：

- 同一用户不能重复收藏同一个专业方向。
- 用户可以为收藏项填写备注。
- 如果专业方向被删除，收藏记录随之删除。

数据表：

- `user_favorite_program`
- `program`

验收标准：

- 用户可以收藏推荐列表或详情页中的专业方向。
- 重复收藏不会产生多条记录。
- 用户可以查看自己的收藏列表。

### 7.5.2 取消收藏

功能说明：

用户可以取消已经收藏的专业方向。

验收标准：

- 取消收藏后，收藏列表不再显示该专业方向。
- 用户不能删除其他用户的收藏。

### 7.5.3 收藏列表

功能说明：

用户可以查看自己收藏过的专业方向。

展示字段：

- 学校名称。
- 学院名称。
- 专业代码。
- 专业名称。
- 学习方式。
- 学位类型。
- 推荐档位，可选。
- 用户备注。
- 收藏时间。

数据表：

- `user_favorite_program`
- `school`
- `college`
- `program`

验收标准：

- 收藏列表只展示当前用户自己的收藏。
- 可以从收藏列表进入学校专业详情。

- `program`

验收标准：

- 管理员可以新增、编辑、停用专业方向。
- 专业方向必须归属于一个学院。
- 同一学院下重复专业方向应受唯一约束限制。

### 7.6.4 科目与专业科目管理

功能说明：

管理员维护初试科目字典，并为专业方向绑定考试科目。

数据表：

- `subject`
- `program_subject`

验收标准：

- 可以维护 101、201、204、301、302、408 等科目。
- 一个专业方向可以绑定多个考试科目。
- 可以按 408 召回候选专业。

### 7.6.5 复试线管理

功能说明：

管理员维护专业方向按年份的复试线。

字段：

- 专业方向。
- 年份。
- 总分线。
- 政治线。
- 英语线。
- 数学线。
- 专业课线。
- 数据来源。
- 可信度。

数据表：

- `admission_score`
- `data_source`

验收标准：

- 同一专业方向同一年只能有一条主要复试线记录。
- 复试线可以绑定数据来源。

### 7.6.6 招生计划管理

功能说明：

管理员维护专业方向按年份的招生计划和复试人数。

数据表：

- `admission_plan`
- `data_source`

验收标准：

- 可以维护计划招生人数、统考名额、推免人数、一志愿复试人数。
- 可以绑定数据来源。

### 7.6.7 拟录取统计管理

功能说明：

管理员维护专业方向按年份的拟录取统计数据。

数据表：

- `admission_result`
- `data_source`

验收标准：

- 可以维护录取人数、拟录取最低分、拟录取平均分、拟录取最高分。
- 可以维护一志愿录取人数和调剂录取人数。
- 可以维护复录比。

### 7.6.8 数据来源管理

功能说明：

管理员维护每条关键数据的来源证据。

字段：

- 来源类型。
- 来源标题。
- 来源 URL。
- 发布日期。
- 文件哈希。
- 备注。
- 可信度状态。

数据表：

- `data_source`

验收标准：

- 关键数据可以追溯到来源。
- 第三方来源必须标记为第三方或待核验。
- 数据冲突时可以标记冲突状态。

### 7.6.9 数据完整度管理

功能说明：

管理员查看和维护某专业方向某年份的数据完整度。

数据表：

- `program_year_data_quality`

验收标准：

- 可以看到某专业方向某年份是否有复试线、招生计划、拟录取统计和官网来源。
- 可以维护完整度等级 A、B、C、D、E。
- 推荐系统可以读取完整度等级。

### 7.6.10 数据采集任务管理

功能说明：

管理员维护待补数据任务，为后续人工采集或 Agent 采集做准备。

数据表：

- `data_collection_task`

验收标准：

- 可以创建缺失复试线、招生计划、拟录取数据等任务。
- 可以维护任务状态。
- 可以设置任务优先级。

## 7.7 Agent 预留需求

### 7.7.1 Agent 不作为 MVP 推荐决策核心

MVP 阶段推荐结果由规则引擎产生。Agent 不直接决定某学校是否推荐，避免早期结果不可控。

### 7.7.2 Agent 二期能力

后续 Agent 可以承担以下工作：

- 根据学校名称和专业方向搜索公开招生资料。
- 解析 PDF、Excel、HTML 或图片中的复试线和拟录取数据。
- 将抽取结果写入 `staging`。
- 对某个推荐结果生成自然语言解释。
- 根据用户画像生成择校报告草稿。
- 提醒管理员哪些学校数据缺失。
- 回答用户关于推荐结果的追问。

### 7.7.3 Agent 数据安全边界

业务规则：

- Agent 抽取结果不能直接写入主库。
- Agent 输出必须进入 `staging` 或待审核状态。
- 管理员审核通过后才能写入复试线、招生计划或拟录取统计主表。
- Agent 推荐解释必须基于系统已有数据和来源，不能编造分数或招生人数。

相关表：

- `staging`
- `data_collection_task`
- `data_source`
- `program_year_data_quality`

- `app_user`
- `user_profile`
- `user_target_program`
- `user_favorite_program`

数据使用原则：

- 第三方数据可以作为线索，但需要标记来源类型。
- 用户画像和推荐记录只能由用户本人或管理员在合理权限下访问。

## 10. 非功能需求

### 10.1 易开发性

项目由单人开发，功能设计应优先保证可实现、可维护、可逐步扩展。

要求：

- 模块边界清晰。
- 优先使用 Spring Boot 常规分层结构。
- 避免一开始引入复杂微服务。
- 避免过度抽象。

### 10.2 数据可信度

要求：

- 关键招生数据必须有来源或可信度状态。
- 缺失数据必须明确展示。
- 冲突数据不能直接用于推荐，需要人工处理。

### 10.3 隐私与安全

要求：

- 密码哈希存储。
- 用户只能访问自己的画像、收藏和推荐记录。
- 管理后台需要管理员权限。
- 用户画像属于个人信息，应避免无关展示。

### 10.4 可扩展性

要求：

- 第一阶段支持 408 计算机方向。
- 数据模型保留扩展到 MPAcc、法硕、医学、教育学、金融等方向的能力。
- 推荐规则应支持按考试类型或专业方向扩展。
- Agent 能力后续接入时，不破坏现有规则推荐主流程。

### 10.5 可部署性

MVP 阶段可先本地运行，后续可部署到服务器供同学朋友访问。

建议：

- 后端 Spring Boot。
- MySQL 8.x。
- 前端 Vue。
- 后续使用 Docker Compose 部署。

## 11. 技术约束

后端建议：

- Spring Boot。
- Spring Security 或简单 JWT 登录认证。
- MyBatis-Plus。
- MySQL 8.x。
- Lombok。

前端建议：

- Vue 3。
- Element Plus。
- Axios。

开发策略：

- 先完成后端核心接口和数据库联通。

## 13. 验收标准

### 13.1 用户端验收

- 用户可以完成注册、登录、退出。
- 用户可以填写并修改画像。
- 用户可以选择目标地区和目标专业。
- 用户可以生成推荐结果。
- 推荐结果包含学校、学院、专业方向、推荐档位和风险提示。
- 用户可以查看学校专业详情。
- 用户可以收藏和取消收藏专业方向。
- 用户可以查看历史推荐记录。

### 13.2 管理后台验收

- 管理员可以登录后台。
- 管理员可以维护学校、学院、专业方向。
- 管理员可以维护复试线、招生计划和拟录取统计。
- 管理员可以维护数据来源。
- 管理员可以查看或维护数据完整度。
- 管理员可以查看或创建数据采集任务。

### 13.3 推荐逻辑验收

- 用户不接受非全时，非全方向不会进入推荐结果。
- 用户选择目标地区后，非目标地区学校不会进入推荐结果。
- 用户预估分高于有效难度分时，推荐档位符合规则。
- 数据不足的专业方向应标记为数据不足。
- 招生人数少、拟录取最低分偏高或数据可信度低时，应出现风险提示。
- 每次推荐应保存到 `recommendation_log`。

### 13.4 数据验收

- 学校、学院、专业方向关系正确。
- 专业方向可以绑定考试科目。
- 招生数据可以绑定数据来源。
- 收藏记录保存到 `user_favorite_program`。
- 历史推荐记录保存到 `recommendation_log`。

## 14. 二期规划

MVP 跑通后，可以继续扩展：

- Agent 辅助资料采集。
- Agent 生成推荐解释。
- Agent 生成个性化择校报告。
- PDF 或 Excel 报告导出。
- 用户反馈入口。
- 付费报告。
- 更多专业方向规则模板。
- 更多地区和学校数据。
- 数据冲突处理工作台。
- 隐私政策和用户协议页面。

## 15. 结论

本 MVP 的目标不是一次性做完整平台，而是先完成一个可以真实使用的择校推荐闭环：

```text
管理员维护数据
-> 用户填写画像
-> 系统规则推荐
-> 用户查看详情
-> 用户收藏方向
-> 系统保存历史记录
```

该闭环适合单人开发，也适合给同学朋友试用。第一版应优先保证数据结构清楚、推荐规则可解释、来源证据可追溯。Agent 能力应作为后续增强，而不是 MVP 的前置依赖。
