# 考研择校推荐助手 — 项目技术文档

> 最后更新：2026-06-18

---

## 1. 项目目标

| 项 | 说明 |
|---|------|
| **项目名称** | AI 考研择校推荐助手（408 统考择校工作台） |
| **解决什么问题** | 408 计算机统考考生面对全国数百所招生院校，信息搜集成本极高。系统用规则筛选 + AI 推荐两套引擎，帮考生从 145+ 所学校中快速圈定目标 |
| **目标用户** | 参加 408 计算机统考的考研学生 |
| **核心价值** | 降低择校信息搜集成本；用数据驱动替代凭感觉选学校；AI 对话式交互解释推荐理由 |

---

## 2. 当前功能

### ✅ 已实现

- √ 用户注册/登录（手机号或邮箱 + 密码，JWT 认证）
- √ 学生画像管理（预估分、目标地区、本科层次、跨考情况、风险偏好）
- √ 规则筛选（按预估分、考试组合、地区、专业方向快速筛选院校）
- √ 筛选结果对比（多所学校并排对比核心指标）
- √ 学校/专业查询（按关键词、地区、档位搜索）
- √ 分数线查询（复试线、录取最低/均分/最高分、三年趋势）
- √ 招生人数查询（招生计划、推免名额、统考名额、复试人数、录取人数）
- √ AI 推荐工作台（三档推荐：冲刺/稳妥/保底，流式生成）
- √ AI 对话（解释推荐理由、对比学校、分析风险、手动调整草稿）
- √ Tool Calling（查询、搜索、对比、添加/移除/替换/填充候选学校）
- √ 推荐报告生成（含执行摘要、档位分析、风险警告、数据声明）
- √ 学校收藏（登录后同步）
- √ Admin 后台管理（数据采集任务、暂存审核、数据质量标注）

### □ 进行中

- □ RAG 知识库（黑名单、招生简章、复试经验）

### □ 规划中

- □ Agent 工作流（多步自主决策）
- □ 推荐报告导出 PDF

## 3. 技术栈

## 5. Tool Calling 设计

AI 对话使用 LangChain4j `@Tool` 注解，工具分为两套：**只读查询工具** 和 **草稿写操作工具**。

### 5.1 只读查询工具（V2ChatTools）

| Tool | 签名 | 功能 |
|------|------|------|
| **getProgramDetail** | `(long programId) → String` | 查询指定 programId 对应学校的完整详情，包含三年趋势数据（复试线、录取均分、最低/最高分、录取人数） |
| **searchPrograms** | `(String keyword, String tier, String region) → String` | 在候选工作集中搜索学校，支持按关键词/档位/地区过滤。无结果时回退到全国数据库搜索 |
| **getRefillCandidates** | `(String tier, int count) → String` | 获取指定档位的候补填充列表 |
| **comparePrograms** | `(long programId1, long programId2) → String` | 并排对比两所学校的核心指标 |
| **getDraftContext** | `() → String` | 查看当前报告草稿的状态（已选学校数、档位分布、关键信息） |

### 5.2 草稿写操作工具（V2DraftActionTools）

| Tool | 签名 | 功能 |
|------|------|------|
| **removeDraftCandidate** | `(long programId) → String` | 从草稿中移除一所学校，自动触发填充策略 |
| **addDraftCandidate** | `(long programId, String tier) → String` | 将候选加入草稿的指定档位 |
| **confirmRefillCandidate** | `(long programId) → String` | 确认用户选择的填充候选 |
| **replaceDraftCandidate** | `(long removeProgramId, long addProgramId) → String` | 替换草稿中的一所学校 |
| **fillTier** | `(String tier) → String` | 将指定档位填满至目标数量（reach/steady/safe） |
| **batchRemoveDraftCandidates** | `(List<Long> programIds) → String` | 批量移除多所学校 |

### 5.3 工具上下文管理

`V2ChatToolContext` 基于 `ThreadLocal` 管理每个 AI 对话线程的上下文，包含：
- `userId` — 当前用户 ID
- `redis` — Redis 模板（读草稿、对话状态）
- `recommendationMapper` — 数据库查询（降级检索）
- 写操作标记 + 草稿变更标记

### 5.4 工具调用约束（在 System Prompt 中约束）

- 必须基于 `getDraftContext()` 返回的实际数据，禁止编造 programId
- 单次回复最多执行 1 个修改类工具（remove/add/replace/fillTier）
- 替换前必须先调用 `comparePrograms` 做对比分析
- 搜索无结果时应扩大搜索范围或建议用户放宽条件
- 禁止展示内部字段名（如 `avgScoreGap`），必须用中文解释

## 6. Prompt 设计

### 6.1 AI 对话 System Prompt（`chat-system.txt`）

```
你是考研择校助手，负责解释、分析和调整用户的择校草稿。

## 用户画像
{profileContext}

## 当前草稿
{draftContext}

## 你的能力
- 分析候选学校的推荐理由、优势和风险
- 回答用户关于学校、专业、录取难度的具体问题
- 对比两所学校（使用 comparePrograms 工具）
- 搜索候选工作集（使用 searchPrograms）
- 查询学校完整数据（使用 getProgramDetail）

## 调整草稿
- fillTier(tier)：将指定档位填满到目标数量
- removeDraftCandidate(programId)：移除学校，自动触发填充建议
- addDraftCandidate(programId, tier)：手动添加学校
- replaceDraftCandidate(removeId, addId)：替换学校，替换前必须对比分析

## 工具调用规则
1. 所有数据必须来自工具调用或草稿上下文，禁止编造
2. 每次对话必须先调用 getDraftContext 获取最新草稿
3. 单次回复最多执行 1 个修改类工具
4. 替换学校前必须先用 comparePrograms 对比，列出风险后确认意图
5. 移除学校后根据填充策略提供选项
6. 不得在回复中展示内部字段名或原始 JSON
```

### 6.2 AI 选择 Prompt（按档位）

#### 冲刺档（`select-reach.txt`）

```
你是一名考研择校顾问。从以下冲刺档候选列表中，选出最适合用户的学校。

规则：
- 仅从给定列表选择，禁止编造
- 每项必须给出选择理由和风险
- 最多选 3 所

输出 JSON：
[{"programId": 1234, "reason": "...", "risks": ["..."], "pros": ["..."], "cons": ["..."]}]
```

#### 稳妥档（`select-steady.txt`）

```
规则：
- 优先选择名额充裕、与分数匹配度好的学校
- 最多选 4 所
```

#### 保底档（`select-safe.txt`）

```
规则：
- 优先选择分数充裕（差距 >= 15 分）、名额稳定的学校
- 最多选 3 所
```

### 6.3 System Prompt 动态构建

每次 AI 对话轮次都会**重建 System Prompt**，确保草稿状态和用户画像实时同步：

```java
String buildSystemPrompt(Long userId) {
    String template = loadPromptTemplate();              // 从 chat-system.txt 加载
    DraftVO draft = draftService.getDraft(userId);       // 从 Redis 读草稿
    String draftCtx = buildDraftContextText(draft);      // 构建档位文本
    String profileCtx = buildProfileContextText(draft);  // 构建画像文本
    return template
        .replace("{profileContext}", profileCtx)
        .replace("{draftContext}", draftCtx);
}
```

## 7. RAG 规划

### 7.1 当前状态：未实现

系统目前没有 RAG 管线。AI 对话只基于：
- System Prompt 中的画像 + 草稿上下文
- Tool Calling 查询 MySQL 结构化数据
- LLM 自身训练知识

### 7.2 规划

| 阶段 | 内容 | 状态 |
|------|------|------|
| **MVP** | 黑名单文档接入验证链路（1 份 MD 文档） | 技术选型完成，待实施 |
| **扩展** | 招生简章、复试经验、学长评价、导师信息 | 规划中 |
| **高级** | 元数据精确过滤、混合检索（BM25 + Dense）、Cross-Encoder 重排序 | 远期 |

### 7.3 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| Embedding 模型 | BGE-M3（智源 BAAI） | 中文 MTEB Top3，1024 维，Ollama 本地运行 |
| 向量数据库 | Redis Stack（RediSearch 模块） | 复用已有 Redis，<10ms 检索延迟 |
| 文档解析 | Apache Tika | 一个依赖支持 MD/PDF/TXT/HTML |
| 框架 | LangChain4j Easy RAG | 与现有 AI 框架统一 |

### 7.4 架构设计

```
文档 → Tika 解析 → 递归分割(500/100) → BGE-M3 向量化 → Redis Stack
                                                              ↓
用户提问 → BGE-M3 向量化 → Redis ANN 检索 → Top-3 片段 → 拼入 System Prompt → LLM 回答
```

扩展现有 `AiChatServiceImpl.buildSystemPrompt()`：在 system prompt 前面拼入检索到的知识片段。

## 8. 推荐逻辑

### 8.1 两套引擎

系统有两套独立的推荐引擎，服务不同场景：

| | V1 规则筛选 | V2 AI 推荐 |
|---|-----------|----------|
| **入口** | `/recommend` 页面 | `/ai-recommend` 页面 |
| **交互** | 填表 → 点按钮 → 看结果 | 填画像 → AI 流式生成 → 对话调整 |
| **核心** | SQL 查询 + Java 排序 | LLM 阅读理解 + 工具调用 |
| **代码** | `ProgramRecommendationServiceImpl` | `AiSelectorServiceImpl` + `DraftServiceImpl` |

### 8.2 V1 规则筛选流程

```
用户输入: 预估分=300, 考试=11408, 地区=[浙江,福建], 专业=[085404]

① SQL 查询 (RecommendationMapper.selectCandidates)
   WHERE: status=active, is_408=1, subject_codes='101,201,301,408'
          province IN ('浙江','福建'), program_code='085404'
   JOIN:  program → college → school (9表联查)
   LIMIT: 300

② Java 过滤 (ProgramRecommendationServiceImpl)
   - scoreRange 模式: filter(matchesAverageDive) → 均分差距在允许范围内
   - 非 scoreRange: filter(非 insufficient_data)

③ Java 排序 (candidateComparator)
   1. 数据完整度 A > B > C
   2. 均分差距绝对值升序
   3. 录取均分降序
   4. 学校名字母序

④ 返回: RecommendResultVO → 前端 Results.vue 渲染
```

### 8.3 V2 AI 推荐流程

```
用户画像 → 候选池构建 → 三档 AI 并行选择 → 草稿 → AI 对话调整 → 报告

详细链路:
① 候选池构建 (CandidateUniverseServiceImpl)
   - MySQL 查询所有匹配考试组合的学校
   - diversityTrim: 同学校只保留得分最高的专业
   - 按预估分差距分为三档: 冲刺(reach) / 稳妥(steady) / 保底(safe)
   - 每档 20-30 所候选

② AI 并行选择 (AiSelectorServiceImpl)
   - 三档同时调 LLM（CompletableFuture × 3）
   - 每档拼 facts 文本 → 加载对应档位 prompt → 调 DeepSeek → 解析 JSON
   - 冲档: 选 3 所 | 稳档: 选 4 所 | 保底: 选 3 所

③ 校验器兜底 (SelectionValidator)
   - programId 在候选池中？（拦截幻觉）
   - 同档内重复？（去重）
   - 数据等级 C？（剔除）
   - 超过上限？（裁剪）

④ 草稿存储
   - Redis Key: ai:v2:draft:<userId>
   - 结构: { tiers: { reach: [...], steady: [...], safe: [...] } }

⑤ AI 对话调整
   - System Prompt 注入画像 + 草稿上下文
   - 10 个 Tool 可用（查询/添加/移除/替换/填充）
   - SSE 流式输出

⑥ 报告生成 (ReportServiceImpl)
   - 从 Redis 读取草稿 → 组装 ReportVO
   - 含执行摘要、档位分析、每校详细卡片、风险警告
```

### 8.4 AI 推荐的关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| AI 是排序还是阅读理解？ | 阅读理解 | AI 读 20-30 所学校描述，自己判断最合适的，而非打分排序 |
| 档内校数 | 冲 3 / 稳 4 / 保 3 = 10 所 | 满足覆盖面的最小数量 |
| 同学校多专业 | 档内去重（diversityTrim） | 同一档内每个学校只保留一个专业 |
| 跨档出现 | 允许（如"杭师大 稳档+保底档"） | 保留后续加软上限（最多两档） |
| 兜底策略 | 多层 fallback | LLM 挂了 → JSON 解析失败 → 按分数取 Top-N |
| 草稿持久化 | Redis（非 MySQL） | 中间态数据，不需要 ACID，需要快速读写 |
